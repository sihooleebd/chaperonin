What is Chaperonin?

A canvas for
protein pipelines

Chaperonin is a typed, visual workflow engine for computational biology. Drag modules onto a node canvas, wire their inputs and outputs together, and hit Run. The frontend serializes the graph and sends it to a Python backend over WebSocket, which executes each tool in order and streams live logs back.

The type system prevents bad connections — a Structure.PDB output won't connect to a Sequence.FASTA input. Every tool's resource requirements (GPU, memory) are declared upfront.

01

Drag modules

Pull any tool from the palette onto the canvas. Each module exposes typed input and output handles. Tools are grouped by category: design, predict, refine, render, convert.

02

Wire connections

Drag from an output handle to a compatible input handle. Chaperonin enforces type compatibility — it won't let you connect a structure file to a sequence consumer. Unconnected required inputs block Run.

03

Run

Hit Run. Chaperonin serializes the graph to JSON, sends it to the Python orchestrator via WebSocket, and streams node.running, node.progress, and node.log events back in real time.

Capabilities

Built for real pipelines

Types

Typed connections

Every handle declares a type — Structure.PDB, Sequence.FASTA, Visual.PNG. The canvas refuses incompatible connections before you even try to run. Union types (Structure.PDB | Structure.mmCIF) widen what an input accepts.

Streaming

Live log streaming

The backend streams node.log, node.progress, and node.done events over a single persistent WebSocket. Every stdout line from RFDiffusion or Rosetta appears in the log panel as execution happens — no polling.

DSL

Bidirectional DSL

The canvas auto-generates a human-readable pipeline description. Edit the DSL text and the canvas re-renders. Edit the canvas and the DSL regenerates. Both representations stay in sync at all times.

Resources

Resource-aware

Each module declares its hardware requirements upfront — GPU count, VRAM, CPU cores, memory. The orchestrator knows what RFDiffusion needs (1× A100, 24 GB) before it schedules anything. No surprises mid-run.

Control

One-click cancel

Send a {"type":"cancel"} message at any point during execution. The backend stops the active node gracefully and emits node.cancelled for any remaining queued nodes. The UI unlocks immediately.

Replay

Serializable graphs

Every run serializes the complete pipeline to JSON — node IDs, module params, edge wiring, and the DSL string. Re-run any past pipeline exactly, share it with a collaborator, or use it as a starting point for the next experiment.

WebSocket protocol

Execution you can watch

When you hit Run, the frontend serializes the graph and sends it to ws://localhost:8000/ws. The Python orchestrator traverses the edge list to determine execution order, then streams events back as each node progresses.

Every run ends with exactly one terminal event — pipeline.done or pipeline.error. Without it the UI stays in running state. The protocol is intentionally simple: one WebSocket, one JSON object per line.

pipeline.startEmitted once before any node events. Carries the total node count.
node.queuedNode is waiting to be scheduled.
node.runningNode has started executing on the backend.
node.progressDrives the progress bar. Carries current and total.
node.logOne line of stdout/stderr from the running tool.
node.doneNode finished successfully.
pipeline.doneTerminal. Pipeline complete.
